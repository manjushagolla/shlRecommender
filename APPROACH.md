# SHL Assessment Recommender — Approach Document

## System Design

### Architecture Overview
A stateless FastAPI service with two endpoints (`GET /health`, `POST /chat`) backed by a FAISS semantic retrieval layer and a conversational LLM agent. Every `/chat` call is self-contained — the full conversation history is passed in and no session state is stored server-side.

### Catalog Ingestion
The SHL Individual Test Solutions catalog was scraped from `shl.com/solutions/products/product-catalog/` using a session-based scraper with retry logic and polite delays. Each of the 377 assessments was enriched with: name, URL, test types (A/B/C/D/E/K/P/S), job levels, description, duration, languages, and a keyword list generated by pattern-matching rules. An `embedding_text` field was pre-built per item combining all metadata fields.

### Retrieval (FAISS)
Each assessment was embedded using `all-MiniLM-L6-v2` (sentence-transformers, 384-dim, ~80MB). Embeddings are normalized and stored in a `faiss.IndexFlatIP` (exact cosine similarity). Exact search was chosen over approximate (IVF/HNSW) because the catalog has under 500 items — at this scale, exact search is instantaneous and introduces zero quantization error.

At query time, the agent builds a weighted retrieval query from the last 3 user messages (repeating the most recent for emphasis), over-fetches 6× (up to 120 candidates), applies hard test-type filters where inferrable, applies a soft job-level boost (+0.04 per matching level), then passes the top 20 to the LLM.

### Agent Design
The agent is a single-pass LLM call (Groq `llama-3.3-70b-versatile` / Gemini `gemini-2.0-flash`) with a structured system prompt injecting the top-20 catalog candidates as grounded context. This is a deliberate RAG design: the LLM has no access to its parametric knowledge about assessments — it can only cite items in the injected catalog block.

**Four conversational behaviors:**

| Behavior | Trigger | Action |
|----------|---------|--------|
| Clarify | < 2 context signals | Ask ONE focused question, return `recommendations: []` |
| Recommend | Role + seniority/skill known | Return 1–10 grounded assessments with brief rationale |
| Refine | User updates constraints | Rebuild shortlist; full history ensures continuity |
| Compare | "difference between X and Y" | Fetch catalog entries for both, LLM compares from data only |

**Safety guards:**
- Rule-based pre-filter catches off-topic/injection before LLM call (saves latency + tokens)
- Turn counter forces recommendations at turn 6+ to stay within the 8-turn cap
- Every returned URL is overwritten with the catalog's URL — the LLM cannot hallucinate URLs
- Any recommended name not found in the catalog is dropped at parse time

### Evaluation
A local harness (`tests/evaluate.py`) replays 10 conversation traces simulating a realistic user who answers questions from a fact sheet, scores:
- **Recall@10** per trace (fraction of expected assessments in the top-10 shortlist)
- **Schema compliance** on every response
- **Turn cap** (max 8)
- **10 behavior probes** (off-topic refusal, prompt injection, vague query, edit-honoring, etc.)

## What Didn't Work and How It Was Improved

**Issue 1 — Silent LLM errors.** The initial `try/except` in the agent swallowed all provider errors, returning a generic fallback. Fixed by removing the catch-all and letting errors propagate to the FastAPI exception handler, which returns a 500 with a visible message for debugging.

**Issue 2 — Deprecated Gemini SDK.** `google-generativeai` raised `FutureWarning` and credential errors. Migrated to `google-genai` (the new unified SDK) with a fallback import for environments that still have the old package.

**Issue 3 — LLM not grounded.** Early prompts allowed the model to use prior knowledge about assessments, leading to hallucinated URLs and invented products. Fixed by injecting the full catalog block into every system message and adding a post-parse validation step that drops any name not found in `shl_meta.json`.

**Issue 4 — Retrieval quality.** Initial single-message queries produced poor candidates for multi-turn conversations. Fixed by concatenating the last 3 user messages (with the most recent repeated) and inferring test-type/job-level filters from free text using a keyword signal map.

## Stack Justification
- **FastAPI** — async, automatic OpenAPI docs, Pydantic validation matches the spec schema exactly
- **FAISS (flat)** — exact cosine search, zero configuration, correct at any catalog size < 10k
- **sentence-transformers / MiniLM** — runs on CPU, 80MB, fast enough for < 30s latency budget
- **Groq (LLaMA 3.3 70B)** — free tier, ~1s inference, strong instruction-following for JSON output
- **Render (Docker)** — free tier supports Docker deploys, persistent disk for catalog files, 2-min cold start tolerance matches the spec's `/health` requirement
